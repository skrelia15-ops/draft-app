# Odessa launch seed data — design

**Date:** 2026-06-09
**Branch:** `feature/supabase-backend-auth`
**Status:** approved-pending-review

## Problem

After migrating Groups/Explore off mockups onto the Supabase backend, the
catalog tables (`routes`, `groups`) are effectively empty / placeholder, so
the app looks bare. For an Odessa launch we want non-empty default content:

- A catalog of routes ("roads") that read as Odessa rides and render on the
  map in Odessa.
- A few product-owned ("official") groups that exist before any users, so
  Discover is populated and users have groups to join.

Per product framing: **routes are generated by the app itself, not people**;
**groups can later be user-created (like events), but for launch the product
seeds some official ones** and attracts users to them.

## Root cause of the empty Explore (found during design)

The reported symptom — Explore showed **no routes** (with geolocation off) — is
**not** caused by geolocation. Geolocation only chooses the map anchor; the
catalog list does not depend on it. The real cause is an auth-timing race in
`RoutesProvider`:

- `routes` RLS is `for select to authenticated using (true)` — anon reads
  return `[]` (verified live: `GET /rest/v1/routes` with the anon key returns
  HTTP 200 `[]`; the table exists, rows are just RLS-hidden from anon).
- `RoutesProvider` fetches the catalog **once on mount** and has **no**
  `onAuthStateChange` subscription. Providers mount in `app/_layout.tsx`
  *before* `AuthProvider`'s async `getSession()` restores the persisted
  session. So the first `listRoutes()` goes out as **anon → `[]`**,
  `isHydrated` flips to `true`, and it never refetches → Explore is empty on
  every authenticated cold start.
- `GroupsProvider` already does the right thing (subscribes to
  `onAuthStateChange` and reloads on user change), so Groups was empty only
  because **no groups were ever seeded**, not because of the race.

**Fix:** make `RoutesProvider` auth-aware, mirroring `GroupsProvider` — load the
catalog on `onAuthStateChange` (and clear on sign-out) instead of a one-shot
mount fetch. Without this fix, seeding more routes changes nothing.

## Decisions (locked)

- **Routes — light approach.** No geometry stored in the DB. Route polylines
  are generated client-side by `buildRoutePreview` anchored to the user's GPS,
  falling back to a fixed city center. We change the fallback from Manhattan to
  Odessa and reseed the catalog with Odessa-named routes. No `routes` schema
  change.
- **Groups — schema change for official groups.** `groups.owner_id` becomes
  nullable and a `is_official boolean` column is added so the product can own
  groups with no human owner. Trigger + RLS updated accordingly.
- **No `OFFICIAL` UI badge** this iteration (the `is_official` column exists
  for seeding + RLS + Discover ordering only).
- **Migrations applied to the live Supabase project** (`fdjeihhtciooahpteign`)
  so data appears in the running app immediately. The Supabase MCP is **not
  connected in the current session** — application path is therefore either
  (a) the user reconnects the Supabase MCP and I apply via it, or (b) I hand
  over the exact SQL to run in the Supabase Studio SQL editor / `db push`.
  This is an execution step to confirm, not a code blocker.

## Part 0 — Fix `RoutesProvider` (the actual empty-Explore bug)

Rework `lib/routes/RoutesProvider.tsx` to load the catalog in response to auth
state, matching the `GroupsProvider` pattern:

- Subscribe to `supabase.auth.onAuthStateChange`.
- On a user id appearing/changing: set `isHydrated=false`, then `refresh()`.
- On sign-out (`uid == null`): clear `routes` and set `isHydrated=true`.
- Keep `refresh()` exposed for pull-to-refresh.
- Dedupe by last uid (as GroupsProvider does) so token refreshes don't refetch.

This guarantees the catalog loads once the session is restored, so Explore
shows routes on every authenticated start regardless of geolocation.

> Note: an alternative fix is relaxing `routes` RLS to allow `anon` reads
> (it's a public catalog). We do **not** do that — keeping reads
> `authenticated`-only and fixing the provider matches the existing
> `GroupsProvider` pattern and avoids changing the security posture.

## Part 2 — Official groups

### Migration `supabase/migrations/20260609050000_official_groups.sql`

1. `alter table public.groups alter column owner_id drop not null;`
2. `alter table public.groups add column is_official boolean not null default false;`
3. Replace `handle_group_created()` so it only inserts the owner membership row
   when `new.owner_id is not null` (official groups have no owner to add).
4. Tighten the insert RLS so authenticated users cannot create official or
   non-self-owned groups:
   `with check (owner_id = auth.uid() and is_official = false)`.
   Existing update/delete policies key on `owner_id = auth.uid()`, so
   null-owner official groups are immutable from any client (product-managed).
5. Recreate the `groups_with_counts` view (it uses `select g.*`, so it picks up
   `is_official`) and re-apply `security_invoker = true` (set in the hardening
   migration).

### Code — types, mappers, ordering

- `lib/groups/mappers.ts` — add `is_official: boolean` and make `owner_id`
  nullable in `GroupRow`. `rowToGroup` does **not** need to surface it (no UI
  badge), but `ownerId` should tolerate `null` (map to `''` or keep `string`;
  domain keeps `ownerId: string` — map `null → ''`).
- `lib/supabase/database.types.ts` — regenerate (or hand-edit) for the new
  column + nullable `owner_id` on `groups` and `groups_with_counts`.
- `lib/groups/storage.ts::listDiscoverGroups` — order official groups first so
  the seeded groups surface despite starting at 0 members:
  `.order('is_official', { ascending: false }).order('member_count', { ascending: false })`.
- `lib/groups/mappers.test.ts` — update the `GroupRow` fixture(s) for the new
  field so the row↔domain test still compiles and passes.

## Nuances

- Official groups start at **0 members** (no owner membership row), so the UI
  shows "0 riders" until people join. This is intentional ("attract users
  later") — ordering them first keeps them visible.
- Scheduled rides for official groups are **out of scope**: `group_rides`
  requires `created_by → auth.users`, which a seed has no user for.
- Real per-route geometry is **out of scope** (light approach chosen).

## Out of scope

- `OFFICIAL` badge UI.
- Official-group scheduled rides.
- Per-route stored geometry / real Odessa street routing.
- User-created groups flow (already implemented separately).
